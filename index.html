Portfolio Account Implementation Plan

This plan converts portfolio from a business-information object into an account-like publishing identity managed by real user accounts. Users still log in with their own account, but they can publish posts as a portfolio if they are the owner or a moderator of that portfolio.

Decision locked: Portfolio posts are public only Decision locked: Post creation is separate for user and portfolio Decision locked: Feed remains one combined feed for user and portfolio posts Decision locked: Owner and moderator have the same portfolio-content permissions Deferred: DMs to portfolio are not in this phase Goal: Minimum breakage to existing APIs

What Already Exists

  • Portfolio business profile model already exists.
  • Portfolio follow/unfollow APIs already exist.
  • Public portfolio listing/detail APIs already exist.
  • User post system already exists with feed, search, likes, comments, bookmarks.

Main Gap Today

  • Posts are still tied only to UserProfile author.
  • Portfolio owner/moderator membership is not modeled strongly enough.
  • Tagged posts for portfolios do not exist yet.
  • Search/feed/profile responses still assume the publisher is always a user.

End Result

  • Portfolio behaves like a public account.
  • Owner and moderators can publish under portfolio identity.
  • Public users can open portfolio profile, posts, tagged posts, and follow it.
  • Audit still records which real user performed the action.

1. What Can Break During Implementation and What Must Be Fixed

This feature can be implemented safely, but only if it is done as an additive change. If we directly replace old fields and old assumptions, some existing app behaviors will break.

Area What Can Break Why It Breaks Required Fix
Post create / read / update / delete User post APIs may fail or return wrong publisher information Current code assumes post.author is always a UserProfile Keep author for compatibility during rollout and add new publisher fields such as publisherType, publisherUserId, and publisherPortfolioId
Feed Portfolio posts may not appear correctly or may be filtered incorrectly Feed logic currently applies user-follow and user-privacy logic using author Keep one combined feed query, but add portfolio-aware handling and treat portfolio posts as public-only
User profile posts Portfolio posts may show inside personal user profile by mistake Current user-post queries fetch by author Update user profile APIs to return only user-published posts, not portfolio-published posts
Search Search results may miss portfolio posts or filter them incorrectly Search excludes deleted/blocked content based on user author assumptions Make search publisher-aware and add portfolio-specific search behavior and filters
Likes and comments notifications Notifications may go to wrong user or fail for portfolio posts Current notification logic sends to post.author only Use createdBy for audit, but define who receives notifications for portfolio posts, usually owner and moderators or only owner based on business rule
Bookmarks Bookmarked post cards may render wrong actor information Bookmark module populates only post.author Return a normalized publisher object in bookmark responses
Post activities / moderation queue Hide/report/moderation screens may show wrong publisher identity These flows also populate post.author as a user Enrich moderation and activity APIs with publisher-aware response fields
Group post wrappers Posts shared into groups may display incomplete publisher data Group-post relations select and populate only author and mentions Update group post wrappers to also return portfolio publisher data and portfolio tags
Portfolio owner/moderator access Moderators may not get access even after UI is ready Current portfolio logic still checks only portfolio.user in key paths Introduce PortfolioMember and update access checks everywhere portfolio ownership is assumed
Portfolio follow protection Owner/moderator follow rules may behave incorrectly Current self-follow prevention checks only portfolio.user Define whether moderators can follow the portfolio and update the rule accordingly
Portfolio add-on payments and lifecycle Portfolio payment and add-on flows may deny valid users Current add-on services check only direct owner relationship Update payment lifecycle access to support the new portfolio member model where needed
Admin portfolio workflows Admin notifications and verification email logic may miss real operators Admin services still read owner/moderator data from legacy fields Keep legacy fields during transition or update admin flows to read from PortfolioMember
Portfolio tagging FE may allow tagging portfolio but backend may not save it Current post validation supports only mentions for user IDs Add a separate field such as mentionedPortfolioIds and update validation, save logic, retrieval, and tagged-posts APIs

Implementation Safety Rule

  • Do not directly replace old fields first.
  • Add new publisher and portfolio-tag fields first.
  • Keep old fields working during migration.
  • Update read APIs before switching FE to new rendering.
  • Only remove old assumptions after all modules are migrated and tested.
Best practice for proper working: implement this as a compatibility-based rollout, not as a hard replacement. That means old user-post behavior must continue working while portfolio behavior is added beside it.

2. Final Product Behavior

  • User logs in with normal user account only. No separate portfolio login is created.
  • A portfolio can have one owner and multiple moderators.
  • User posts and portfolio posts have separate creation sections, so there is no need to choose publisher inside one composer.
  • Feed is not separated. User posts and portfolio posts both appear in the same main feed.
  • If published as a portfolio, the feed/profile/search must show the portfolio as the publisher.
  • Portfolio posts are always public in this phase. User privacy modes are not used for portfolio posts.
  • Owner and moderators can create, edit, and delete portfolio posts, including posts created by other moderators.
  • Portfolio detail screen gets account-like tabs: Posts, Tagged Posts, About.
  • Users can follow portfolios, but portfolio posts do not require follower-only visibility.
  • Comments and likes remain user-based in this phase. They do not become portfolio-based yet.
DMs to portfolio can be implemented later cleanly. To make that possible, this phase should introduce a proper publisher/actor model now, instead of hardcoding everything to author: UserProfile.

Manage members means adding a moderator, removing a moderator, or changing who is allowed to operate that portfolio. In this proposal, owner and moderator have the same portfolio usage permissions, but only the owner manages the member list.

4. Architecture Decision

Identity Model

  • User account remains the real authenticated actor.
  • Portfolio becomes a public publishing identity.
  • PortfolioMember becomes the source of truth for owner/moderator access.
  • Post publisher becomes polymorphic: user or portfolio.

Why This Is Better

  • Does not require a second login system.
  • Keeps audit trail of which human did the action.
  • Allows future DM-as-portfolio without redesigning again.
  • Minimizes breaking changes by extending current APIs instead of replacing them.

9. Tagging Portfolios: Search and Save Scenarios

If portfolio tagging is part of the feature, then portfolio must also appear in the mention/tag search. The important rule is: user tags and portfolio tags should not be stored in one mixed field.

8.1 Search Behavior

  • When user types @ while writing a post, search results should return both users and portfolios.
  • The FE should show one combined dropdown list, but each result should clearly show its type.
  • Example dropdown rows: John Mathew - User, Dress Hub - Portfolio, Car Zone - Portfolio.
  • This search behavior should be the same in both sections: normal user-post section and portfolio-post section.

8.2 Save Behavior

Selected Tag Type Save Field Purpose
User mentions: [userProfileId] Existing user mention behavior stays the same
Portfolio mentionedPortfolioIds: [portfolioId] Used to show tagged posts inside portfolio profile

8.3 Scenario Examples

  • Scenario 1: A normal user creates a post and tags another user. Save into mentions only.
  • Scenario 2: A normal user creates a post and tags a portfolio. Save into mentionedPortfolioIds only.
  • Scenario 3: A normal user creates a post and tags both a user and a portfolio. Save users in mentions and portfolios in mentionedPortfolioIds.
  • Scenario 4: A moderator creates a post from the portfolio section and tags a user. The post publisher is portfolio, but tagged user still goes into mentions.
  • Scenario 5: A moderator creates a post from the portfolio section and tags another portfolio. The post publisher is portfolio, and tagged portfolio goes into mentionedPortfolioIds.
  • Scenario 6: A public visitor opens a portfolio profile and goes to Tagged Posts. They should see posts where that portfolio ID exists in mentionedPortfolioIds.

8.5 FE Rendering Notes

  • FE should keep one visual search list for tags, but each result must show whether it is a user or portfolio.
  • FE should send selected user IDs to mentions.
  • FE should send selected portfolio IDs to mentionedPortfolioIds.
  • FE should support portfolio tagged-post display on the portfolio public profile.
Recommendation: do not overload the current mentions array with both users and portfolios. Keep two separate fields so notifications, tagged-post queries, and profile rendering stay simple.

11. What FE Team Must Change

  • Keep a separate user posting section and a separate portfolio posting section.
  • Do not create a separate portfolio feed screen unless business asks later; both post types render in the same feed.
  • Render feed/post cards using a generic publisher component instead of assuming user author only.
  • Add portfolio dashboard list using /portfolios/me/managed.
  • Add portfolio public profile tabs: Posts, Tagged Posts, About.
  • Add permission-based actions from backend response: show/hide edit/delete/create buttons.
  • Handle combined tag search dropdown with both users and portfolios.
  • Send user tags to mentions and portfolio tags to mentionedPortfolioIds.
  • Keep likes/comments UI user-based for this phase.

12. What Client Needs to Know

  • This is not creating a second login account; it is creating a business identity managed by existing users.
  • Owner and moderators use the portfolio in almost the same way for normal operations.
  • The only special owner area is manage members, which means adding/removing moderators or changing portfolio access.
  • Portfolio posts are public only in phase 1, which reduces complexity and speeds release.
  • Followers can follow portfolios, but following is not required to view portfolio posts.
  • Users can tag portfolios in posts, and tagged portfolio posts will appear in the portfolio's tagged-posts section.
  • Portfolio DMs are intentionally deferred, not blocked. The architecture proposed here keeps that future open.
  • Comments and likes remain from user accounts in this phase unless business decides to expand scope.

13. Deferred Scope: DMs to Portfolio

Yes, DMs to portfolio can be done later cleanly if we keep the actor model correct now. Current chat collections are still user-only: participants and senders point only to UserProfile.

Future Chat Schema Direction

{
  conversation.participants: [
    {
      participantType: "user" | "portfolio",
      userId: ObjectId | null,
      portfolioId: ObjectId | null
    }
  ],
  message: {
    senderType: "user" | "portfolio",
    senderUserId: ObjectId | null,
    senderPortfolioId: ObjectId | null,
    sentByUserId: ObjectId
  }
}

That future change is independent from the portfolio-post rollout as long as posts are redesigned now around publisher and createdBy.

14. Final Recommendation

The best path is to keep existing user login and portfolio follow features, add a real PortfolioMember model, extend posts with a publisher concept, add portfolio tagged-post support, and keep portfolio posts public-only in V1.

This gives the client the business-account behavior they want, keeps FE changes understandable, avoids a second login architecture, and leaves a clean future path for DMs to portfolio.
